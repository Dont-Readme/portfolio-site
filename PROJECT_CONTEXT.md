# PROJECT_CONTEXT — portfolio-site

## 1. 한 줄 요약

이직 지원 시 이력서와 함께 첨부할 프로젝트 참고용 개인 포트폴리오 웹사이트.

## 2. MVP 목표

- 채용 담당자가 지원자의 포지셔닝과 프로젝트 경험을 빠르게 이해할 수 있게 한다.
- Home / About / Projects / Contact 구조를 제공한다.
- 프로젝트 리스트와 `/projects/{slug}` 상세 페이지를 제공한다.
- 프로젝트 상세 콘텐츠는 MDX/Markdown으로 관리하고, 메타데이터는 `src/data/projects.ts`에서 관리한다.
- 프로젝트 상세는 8개 고정 섹션을 따른다.
- 이력서 다운로드나 Resume 페이지는 제공하지 않는다.
- Framer Motion으로 감각적인 애니메이션을 제공하되 가독성을 해치지 않는다.
- Vercel 무료 URL로 배포 가능한 구조를 만든다.

## 3. 현재 결정된 사항(Decision Log)

1. 프로젝트명은 `portfolio-site`로 한다.
2. 대상 사용자는 채용 담당자로 한다.
3. 메뉴는 `Home / About / Projects / Contact`로 한다.
4. 프로젝트 상세는 `/projects/{slug}` 개별 페이지로 만든다.
5. 프로젝트 상세 기본 섹션은 8개만 사용한다.
6. 프로젝트 본문은 MDX/Markdown으로 관리하고 메타데이터는 TypeScript 데이터 파일에서 관리한다.
7. 백엔드 API와 DB는 사용하지 않는다.
8. 이력서 PDF, Resume 메뉴, 이력서 외부 링크는 제공하지 않는다.
9. 기술 스택은 Next.js + TypeScript + Tailwind CSS + MDX + Framer Motion + Vercel로 한다.
10. Figma remote MCP 사용을 우선 고려한다.
11. AGENTS.md를 생성한다.
12. PLANS.md와 plans/ 스텁 파일을 생성한다.
13. `create-next-app --src-dir` 결과에 맞춰 애플리케이션 코드는 `src/` 아래에 둔다.
14. 로컬 개발 서버는 Windows Codex 런타임의 Turbopack worker 권한 이슈를 피하기 위해 `next dev --webpack`을 사용한다.
15. Home은 검은 메인 영역, 확대 진입하는 흰색 My Work 영역, 검은 Contact 영역이 스크롤로 이어지는 단일 페이지로 구성한다.
16. 우측 상단 `Menu`는 Home, About, Projects만 제공하고, Projects는 Home의 `#my-work` 섹션으로 이동한다.
17. About 페이지와 프로젝트 상세 페이지는 디자인 확정 전까지 빈 페이지로 둔다.
18. 제안서 자동화 프로젝트의 1차 데모는 백엔드/API 없이 RFP Copilot의 페이지 구조를 반영하고, GitHub에 남아 있던 export preview와 planner cache 실행 결과를 화면용 데이터로 옮긴 앱형 데모로 구현한다.
19. 과제 면접용 AI 학습 코치 데모는 `/ai-coach-demo` 직접 경로로만 제공하고 Home/메뉴에는 진입 링크를 추가하지 않는다.
20. AI 학습 코치 데모에 한해 DB 없이 stateless Route Handler를 사용한다. 클라이언트는 허용된 fixture ID만 보내며 OpenAI 환경변수와 호출은 서버에만 둔다.
21. 기획 문서에 구체적인 10명 fixture 정의가 없어, 문서에 명시된 신규/기존 전환·프로필 유무·콘텐츠 선호·후보 부족·다양성 시나리오를 포괄하는 결정론적 합성 fixture 10명을 사용한다.
22. AI 학습 코치의 기준 시점과 합성 데이터·평가 라벨·로직·프롬프트 버전을 고정하고, API 미설정 또는 2회 실패 시 규칙 기반 fallback으로 동일한 검증 흐름을 끝까지 제공한다.
23. AI 학습 코치 평가 라벨은 무작위 순환 패턴을 사용하지 않고, `codex-rubric-judged.v2` 루브릭으로 독립 판정한 사용자-후보 93개 정답(`personalFitGold`, `ndcgRelevance`)을 평가 전용 모듈에 명시적으로 고정한다. 판단 근거는 별도 라벨링 워크북에 보존하며 AI 프롬프트에는 전달하지 않는다.
24. 실제 Structured Outputs 호출이 기존 2초 제한에서 두 번 모두 시간 초과되어 fallback만 표시되므로, 면접 데모의 실 API 확인을 위해 OpenAI 요청 제한을 시도당 20초로 조정하고 최대 2회 재시도·fallback·지연 시간 기록은 유지한다.
25. AI 학습 코치의 학생용 추천 문구는 초등학생에게 맞는 따뜻한 해요체 한 문장으로 작성하고 오답률·취약도 점수 같은 진단 수치는 직접 노출하지 않는다. 정확한 수치는 Evidence와 `evidence_ids`로 분리해 근거 추적성을 유지하며 프롬프트 버전은 `candidate-fit-ko.v2`로 관리한다.
26. AI 학습 코치 데모 화면은 면접관이 계산 흐름을 빠르게 읽을 수 있도록 01~08의 한글 섹션명으로 단순화한다. 장식용 영문 라벨, 합성 데이터 규모·고정 버전·출시 후 KPI 카드, 순위 이동 및 정상 상태 배지는 제거하고 실제 계산표·품질 지표·근거 및 실행 기록은 유지한다.
27. AI 학습 코치의 학생용 추천 문구는 `candidate-fit-ko.v3`부터 80자 이하·최대 두 문장으로 확장한다. 최근 학습·목표·선호 유형·시간·수준/난이도·학습 활동 중 근거가 가장 강한 이유 하나를 먼저 제시하고 유형별 구체 행동을 연결하며, 후보 간 시작말·구조·동사를 다양화한다. 진단 수치는 계속 학생 문구에서 제외하고 Evidence로만 추적한다.
28. `candidate-fit-ko.v4`부터 반복을 유도하던 완성 문장 예시를 제거하고, 정규화한 근거 강도와 0.15 허용 범위로 후보 간 `comment_reason`을 배분한다. 이유·행동 순서를 고정하지 않고 시작·구조·끝맺음과 상투 표현 사용을 응답 전체에서 조정하며, Structured Output과 서버 검증에도 `comment_reason` 및 근거 연결을 포함한다.
29. AI 응답 실패 시 06은 기획대로 규칙 기반 대체 추천을 유지한다. 05의 비교 결과와 07의 선택 사용자 NDCG도 실제 최종 노출 결과를 기준으로 전환하고 대체 상태를 한글로 명시해, 가상의 AI 혼합안과 실제 노출안이 섞여 보이지 않도록 한다.
30. 결정 29를 면접 데모 목적에 맞게 대체한다. API 호출 성공 여부와 무관하게 05~07은 AI 추천이 정상 생성된 시나리오를 보여주며, 실패 시 결정론적 모의 AI 적합도·문구로 AI 혼합 Top 3, 상세 카드, 품질 평가를 일관되게 유지한다. 실제 API 실패·대체 출처는 08 실행 기록에서만 투명하게 확인한다.
31. AI 학습 코치의 역할을 정형 계산과 의미 판단으로 분리한다. 시스템은 유형·시간·수준 적합도를 계산하고, AI는 최대 10개 후보의 `goal_relation`, `comment_reason`, `comment`만 생성한다. 서버는 목표 관계를 점수로 매핑하고 적용 가능한 항목만 정규화하며, 실제 근거 ID 연결·개인 적합도와 취약도의 50:50 결합·다양성 적용을 담당한다. API가 없거나 실패해도 05~07은 `정상 AI 응답 가정 시나리오`로 유지하되 08의 실제 API 상태와 출처는 분리한다. 데모의 시도당 20초·1회 재시도 설정은 운영 기획의 2초 반영 목표와 별도로 관리한다.
32. AI 입력에는 정확한 최근 오답률 대신 `낮음/보통/높음`에 대응하는 구간 강도만 전달한다. 서버 근거 검증은 선택된 이유 유형에 맞는 근거 ID가 카탈로그에 존재하고 중복되지 않는지 확인하는 범위로 명시하며, 문구 의미 정합성은 별도 품질 지표로 평가한다. `live_ai` 표시는 실제 OpenAI 응답에만 사용하고 서버 모의 응답은 `simulated_ai_scenario`로 구분한다.
33. `candidate-fit-ko.v6`부터 학생용 문구를 기계적인 안내가 아닌 친근한 멘토 말투로 조정한다. 예상 시간과 조건은 나열 대신 부담 없는 제안으로 연결하되 입력에 없는 학습 효과는 약속하지 않는다. 최대 10개 후보를 처리하므로 세 가지 고정 템플릿을 배정하지 않고 질문·공감·행동 우선·목표 연결·작은 도전 등 표현 방식을 분산한다. 완성 문장 Few-shot은 표현 앵커링과 반복을 다시 만들 수 있어 사용하지 않는다.

## 6. 중요한 제약/주의사항

- Resume 페이지를 만들지 않는다.
- 이력서 PDF 다운로드를 만들지 않는다.
- 이력서 외부 링크를 만들지 않는다.
- 로그인, DB, 백엔드 API, 문의 폼을 만들지 않는다.
- 프로젝트 상세 섹션은 아래 8개를 기본으로 한다.
  1. 프로젝트 한 줄 요약
  2. 배경 / 문제
  3. 목표
  4. 내 역할
  5. 기획 과정
  6. 주요 화면 또는 이미지
  7. 핵심 기능
  8. 결과 / 성과
- 애니메이션은 정보 탐색과 가독성을 방해하면 안 된다.
- `.env`는 커밋하지 않고 `.env.example`만 커밋한다.
- 외부 링크는 새 탭으로 열고 `rel="noreferrer"`를 포함한다.
